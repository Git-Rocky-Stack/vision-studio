## Summary

Vision Studio has a strong dark-cinema design foundation with well-defined color tokens, cinematic animations, and a cohesive visual identity. However, the codebase suffers from **critical accessibility gaps across nearly every component**, **widespread hardcoded values that bypass the design token system**, **missing interaction states** (loading, error, disabled, empty), and **performance issues** from lack of memoization. The UI is approximately 60-65% aligned with production-grade standards.

## Critical Issues

### Issue 1: Missing ARIA Attributes Across All Interactive Elements

**Severity:** Critical
**Location:** All 46 component files
**Category:** Usability / Accessibility

**Problem:**
Nearly every interactive element in the application is missing required ARIA attributes. Buttons lack `aria-label` (especially icon-only buttons), toggle buttons lack `aria-pressed` or `aria-checked`, expandable sections lack `aria-expanded`, tabs lack `role="tab"` / `role="tablist"` / `aria-selected`, modals lack `role="dialog"` / `aria-modal`, and menus lack `role="menu"` / `role="menuitem"`. The Sidebar navigation has no `aria-current="page"` on the active item.

**Impact:**
Screen reader users cannot operate the application. Fails WCAG 2.1 Level A (4.1.2 Name, Role, Value).

**Recommendation:**
Systematically audit every interactive element and add appropriate ARIA attributes. Priority files:

- `Sidebar.tsx`: Add `aria-current="page"` to active nav item, `aria-label` to collapse button
- `ToolStrip.tsx`: Add `aria-pressed` to active tool, `aria-label` to all buttons
- `EditPropertiesPanel.tsx`: Add `role="tablist"`, `role="tab"`, `role="tabpanel"`, `aria-selected`
- `ControlNetPanel.tsx`, `AIToolsPanel.tsx`, `TextControls.tsx`: Add `aria-checked` to all toggle switches
- `ModelSelector.tsx`: Add `aria-haspopup="listbox"`, `role="listbox"`, `role="option"`
- `PromptHistory.tsx`: Add `role="dialog"`, `aria-modal="true"`
- `CanvasContextMenu.tsx`: Add `role="menu"`, `role="menuitem"`, `aria-disabled`
- `ImagePreviewModal.tsx`: Add `aria-label` to navigation/close buttons

**Code Example:**

```tsx
// Before (Sidebar.tsx)
<button onClick={() => setActivePanel(item.id)} title={item.label}>
  <item.icon className="w-5 h-5" />
</button>

// After
<button
  onClick={() => setActivePanel(item.id)}
  aria-label={item.label}
  aria-current={activePanel === item.id ? 'page' : undefined}
>
  <item.icon className="w-5 h-5" aria-hidden="true" />
</button>
```

### Issue 3: Slider Component Missing role="slider" and Keyboard Support

**Severity:** Critical
**Location:** `src/components/ui/Slider.tsx:1-116`
**Category:** Usability / Accessibility

**Problem:**
The custom Slider component uses raw `<div>` elements with pointer events but lacks `role="slider"`, `aria-valuemin`, `aria-valuemax`, `aria-valuenow`, `aria-label`, and keyboard arrow key support. This component is reused across 10+ locations (generation params, LoRA weights, filter intensity, crop, text controls, AI tools).

**Impact:**
All slider-based controls are completely inaccessible to keyboard and screen reader users. This is the highest-impact single component fix.

**Recommendation:**

```tsx
// Before
<div ref={trackRef} className="relative h-5 flex items-center cursor-pointer" ...>

// After
<div
  ref={trackRef}
  role="slider"
  tabIndex={0}
  aria-valuemin={min}
  aria-valuemax={max}
  aria-valuenow={value}
  aria-label={label}
  onKeyDown={(e) => {
    const step = (max - min) / 100;
    if (e.key === 'ArrowRight' || e.key === 'ArrowUp') onChange(Math.min(max, value + step));
    if (e.key === 'ArrowLeft' || e.key === 'ArrowDown') onChange(Math.max(min, value - step));
  }}
  ...
>
```

### Issue 4: Toggle Switches Are Not Semantic HTML

**Severity:** Critical
**Location:** `SettingsPanel.tsx:269-284`, `ControlNetPanel.tsx:56-72`, `AIToolsPanel.tsx` (3x), `TextControls.tsx`, `LayerPanel.tsx`
**Category:** Usability / Accessibility

**Problem:**
All toggle switches are built with `<div>` + `<span>` elements styled as switches. They lack `role="switch"`, `aria-checked`, and keyboard operability (Space key to toggle). This pattern appears in 8+ locations.

**Impact:**
Screen readers announce these as generic elements. Users cannot determine toggle state or operate them with keyboard.

**Recommendation:**

```tsx
  role="switch"
  aria-checked={enabled}
  aria-label="Enable ControlNet"
  onClick={() => setEnabled(!enabled)}
  className={cn('w-9 h-5 rounded-full transition-colors', enabled ? 'bg-red-primary' : 'bg-void')}
>
  <span className={cn('block w-4 h-4 rounded-full bg-white transition-transform', enabled ? 'translate-x-4' : 'translate-x-0.5')} />
</button>
```

### Issue 5: Color-Only GPU Status Indicator (Color Blindness Failure)

**Severity:** Critical
**Location:** `Sidebar.tsx:111-127`
**Category:** Usability / Accessibility

**Problem:**
GPU status uses green (`bg-green-500`) vs. yellow (`bg-yellow-500`) to distinguish "connected" from "available but not connected." These colors are not in the design token system and rely solely on color to convey meaning, which fails for color-blind users (~8% of male users).

**Impact:**
Color-blind users cannot determine GPU status. Fails WCAG 1.4.1 (Use of Color). Also breaks design system consistency since green/yellow are not defined tokens.

**Recommendation:**
Add a text label or icon indicator alongside the color, and move colors to design tokens:

```tsx
// Add icon + text indicator
{systemInfo.comfyuiConnected ? (
  <><CheckCircle2 className="w-3 h-3" /> Connected</>
) : (
  <><AlertCircle className="w-3 h-3" /> Disconnected</>
)}
```

### Issue 6: EditCanvas Drawing Performance - Array Copy on Every Mouse Move

**Severity:** Critical
**Location:** `EditCanvas.tsx:140`
**Category:** Performance

**Problem:**
During freehand drawing, `handleMouseMove` creates a full array copy on every mouse movement event:
```tsx
setLines([...lines.slice(0, -1), lastLine]);
```
This creates a new array with `concat()` on every point, leading to O(n) allocations per mouse event at 60+ events/second.

**Impact:**
Drawing will become increasingly laggy as lines accumulate. On complex drawings, this can cause visible jank and dropped frames.

**Recommendation:**
Use a mutable ref for in-progress drawing and only commit to state on `mouseUp`. Or use a drawing library like `react-konva`'s built-in line tool with proper batching.

### Issue 9: No Focus Trap in Modals

**Severity:** Critical
**Location:** `ImagePreviewModal.tsx`, `PromptHistory.tsx`, `TemplatePreviewModal.tsx`, `TemplateCreator.tsx`
**Category:** Usability / Accessibility

**Problem:**
Modal dialogs do not trap keyboard focus. When a modal is open, Tab can move focus to elements behind the modal. No focus is restored to the triggering element when the modal closes.

**Impact:**
Keyboard users can interact with hidden content behind modals. Fails WCAG 2.4.3 (Focus Order).

**Recommendation:**
Use a focus-trap library (e.g., `focus-trap-react`) or implement manual focus trapping:
```tsx
useEffect(() => {
  const previousFocus = document.activeElement as HTMLElement;
  modalRef.current?.focus();
  return () => previousFocus?.focus();
}, []);
```

### Issue 10: No Loading/Error States for Image Display

**Severity:** Critical
**Location:** `Canvas.tsx:46-56`, `AssetsPanel.tsx:260-264`, `ResultCard.tsx`, `GenerationQueue.tsx`
**Category:** Usability

**Problem:**
Images are loaded without any loading skeleton, spinner, or error handling. In `Canvas.tsx`, `new Image()` has no `onerror` handler - if the image fails to load, the canvas shows nothing with no error message. Asset thumbnails have no lazy loading or placeholder. ResultCard images have no loading state.

**Impact:**
Users see blank spaces with no indication whether content is loading, failed, or empty. Poor UX on slow disks or large files.

**Recommendation:**
Add a shared `<ImageWithFallback>` component that handles loading, error, and empty states:
```tsx
function ImageWithFallback({ src, alt, ...props }) {
  const [state, setState] = useState<'loading' | 'loaded' | 'error'>('loading');
  return (
    <>
      {state === 'loading' && <Skeleton />}
      {state === 'error' && <ErrorPlaceholder />}
      <img src={src} alt={alt} onLoad={() => setState('loaded')} onError={() => setState('error')} {...props} />
    </>
  );
}
```

### Issue 11: Hardcoded `text-[10px]` and `text-[9px]` Font Sizes Break Type Scale

**Severity:** Critical
**Location:** `Timeline.tsx` (4x), `GeneratePanel.tsx:593,599`, `PromptHistory.tsx`, `LoRAMixer.tsx:122`, `EditPropertiesPanel.tsx`, `AIToolsPanel.tsx`, `ResultCard.tsx:69`, `ResultsGrid.tsx`, `TemplateCreator.tsx:313`, `TemplatePreviewModal.tsx:178`, `LayerPanel.tsx`, `BatchPanel.tsx:334`

```css
}
```

**Severity:** Major
**Location:** All component files
**Category:** Performance

**Problem:**
None of the 46 components use `React.memo`. Parent state changes (e.g., in `App.tsx` or `WorkspaceLayout.tsx`) cause cascading re-renders of the entire component tree. This is especially problematic for Canvas, Timeline, and effects components (FilmGrainOverlay, AmbientParticles).

Wrap stable components in `React.memo`, starting with: `Sidebar`, `Header`, `Canvas`, `Timeline`, `ToolStrip`, `FilmGrainOverlay`, `AmbientParticles`, all page panels.

### Issue 16: Spacing Inconsistency Across the Entire Application

**Severity:** Major
**Location:** Multiple files
**Category:** Visual Design

**Problem:**
Spacing values are inconsistent and frequently break the 8pt grid:
- `py-2.5` (10px) appears in 8+ locations
- `px-3 py-2.5` is the most common button padding (12px x 10px - not 8pt aligned)
- `gap-1.5` (6px), `gap-0.5` (2px) used alongside `gap-2` (8px) and `gap-3` (12px)
- `my-1.5` (6px), `pb-1` (4px), `p-1.5` (6px) scattered throughout

**Recommendation:**
Define a spacing scale in the design system and enforce it:
```
4px  (0.25rem) - micro spacing (icon gaps only)
8px  (0.5rem)  - compact spacing
12px (0.75rem) - standard gap
16px (1rem)    - section spacing
24px (1.5rem)  - large spacing
32px (2rem)    - section breaks
```

### Issue 19: Canvas useEffect Image Loading Has No Cleanup or Error Handling

**Severity:** Major
**Location:** `Canvas.tsx:46-56`
**Category:** Performance / Code Quality

**Problem:**
`useEffect` creates `new Image()` on every `currentImage` change without cleanup. If the component unmounts during load, the callback still fires. No `onerror` handler means silent failures.

**Recommendation:**
```tsx
useEffect(() => {
  if (!currentImage) return;
  let cancelled = false;
  const img = new Image();
  img.onload = () => { if (!cancelled) setImageSize({ width: img.width, height: img.height }); };
  img.onerror = () => { if (!cancelled) setImageError('Failed to load image'); };
  img.src = currentImage;
  return () => { cancelled = true; };
}, [currentImage]);
```
